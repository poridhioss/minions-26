# Customer Churn MLOps Pipeline

## Introduction

This lab builds a production-shaped MLOps pipeline for predicting customer churn. The system covers the full lifecycle: data generation, feature preprocessing, model training, experiment tracking, model serving, and a user-facing web UI. The deliverable is a small but complete system that demonstrates each stage of an MLOps workflow in code you can read end-to-end.

The lab follows a structured approach. Each chapter introduces one component, explains the reasoning behind it, and connects it to the production problem the system solves. The final chapter assembles the parts into a single running system you can demo to a stakeholder.

**Prerequisites:** Basic Python knowledge, familiarity with command-line tools, and conceptual exposure to REST APIs and supervised learning.

## Learning Objectives

By the end of this lab, you will be able to:

1. Implement a reproducible training pipeline with Scikit-learn
2. Track experiments using MLflow and compare model runs
3. Build a prediction service with FastAPI and Pydantic validation
4. Containerize a model service with Docker
5. Create a frontend UI that consumes a model API
6. Diagnose and resolve common MLOps integration issues

## Prologue: The Challenge

You join the customer analytics team at a retail bank. The product team wants to identify customers likely to close their accounts within the next quarter so retention specialists can intervene early. The team has historical customer data but no production model.

Your task is to build an end-to-end ML system that:

- Loads customer data and trains at least two candidate models
- Tracks every experiment so results are auditable and comparable
- Exposes predictions through an HTTP API that downstream systems can call
- Provides a web interface that business users can use without writing code
- Packages the service so deployment is a single command

Success criteria: a stakeholder can request predictions for individual customers or upload a CSV, and the system returns churn probabilities with supporting evidence of how the model was built and evaluated.

## Chapter 2: Preprocessing

Raw data is never directly consumable by a model. Numeric features must be scaled, categorical features encoded, and the same transformations applied at training and inference time. This chapter builds the preprocessing layer.

### 2.1 What You Will Build

A `ColumnTransformer` that scales numeric features and passes binary features through unchanged.

### 2.2 Think First: Train-Serve Skew

**Question:** A model is trained with `StandardScaler` applied to the `age` column. At inference, the API receives a row and passes `age` directly to the model. What goes wrong?

<details>
<summary>Click to review</summary>

The model expects scaled values, so the predictions will be wrong. This is a classic bug called *train/serve skew*. The fix is to persist the fitted scaler and apply it to every incoming row before prediction.

</details>

### 2.3 Implementation

The preprocessing logic lives in `src/preprocess.py`. Review the file to understand the structure:

- `FEATURE_COLUMNS`: the canonical list of features in a fixed order
- `NUMERIC_COLUMNS` and `CATEGORICAL_COLUMNS`: split used by the transformer
- `get_preprocessor()`: returns a fitted `ColumnTransformer`
- `preprocess_for_training()`: end-to-end helper for the training script

### 2.4 Understanding the Code

The `ColumnTransformer` is wrapped in a `Pipeline` along with the model at training time. This means the saved artifact contains the fitted preprocessor. At inference, the pipeline receives raw rows and applies scaling internally. This is how the system avoids train/serve skew.

## Chapter 3: Model Training and Tracking

Training a single model produces a single result. Production teams train many models with different hyperparameters and need a way to compare them. This chapter introduces MLflow as the experiment-tracking layer.

### 3.1 What You Will Build

A training script that trains two candidate models (Logistic Regression and Random Forest), logs every run to MLflow, evaluates each on a held-out test set, and saves the best model to disk.

### 3.2 Think First: Selecting the Best Model

**Question:** Accuracy on an imbalanced dataset is 89% if the model predicts "no churn" for every customer. Why is accuracy a poor metric here?

<details>
<summary>Click to review</summary>

A model that never predicts churn achieves 89% accuracy by exploiting class imbalance, but it has zero business value. Metrics like F1, precision, and recall expose this failure mode. This lab uses F1 as the selection criterion because it balances false positives and false negatives.

</details>

### 3.3 Implementation

## Chapter 4: Building the Prediction API

A trained model on disk is not yet a service. Applications need an HTTP interface to send customer data and receive predictions. This chapter builds that interface with FastAPI.

### 4.1 What You Will Build

A FastAPI application with three endpoints:

| Method | Endpoint | Purpose |
|--------|----------|---------|
| GET | `/health` | Liveness probe and model status |
| POST | `/predict` | Score one or more customers |
| POST | `/train` | Retrain models and hot-reload the predictor |

### 4.2 Think First: Validation at the Boundary

**Question:** A client sends a POST to `/predict` with `age: -5`. What status code should the API return, and why?

<details>
<summary>Click to review</summary>

The API should return `422 Unprocessable Entity`. The age is structurally valid (it is an integer) but semantically invalid (negative). Pydantic enforces the `ge=18` constraint declared in `CustomerFeatures` and rejects the request before it reaches the model.

</details>

### 4.3 Implementation

Start the API server:

```powershell
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

Open `http://localhost:8000/docs` in a browser. FastAPI generates interactive Swagger documentation automatically from the Pydantic schemas.

### 4.4 Understanding the Code

Review `app/main.py` and locate the following:

- **Lifespan hook**: loads the model into memory once at startup
- **`/health` endpoint**: reports whether the model is loaded
- **`/predict` endpoint**: validates input, calls the predictor, returns results
- **`/train` endpoint**: retrains models and swaps the in-memory predictor

The Pydantic schemas in `app/schemas.py` define the request and response shapes. Validation errors are returned as `422` with a descriptive message.

## The Principles

1. **Persist the full pipeline** — Saving only the model weights is not enough. The fitted preprocessor must travel with the model so inference reproduces training exactly.
2. **Validate at the boundary** — Bad input should be rejected at the API layer (Pydantic) before it reaches business logic. Return `422` for schema errors, `400` for business-rule violations.
3. **Track every experiment** — MLflow is not optional. Without experiment tracking, model selection becomes folklore. Log parameters, metrics, and artifacts for every run.
4. **Decouple the model from the interface** — The API owns the contract. Multiple frontends (Streamlit, CLI, mobile app) can consume the same service.
5. **Make the run reproducible** — Fixed random seeds, pinned dependencies, and a deterministic data generator mean anyone can reproduce the results.
6. **Choose metrics that match the problem** — Accuracy is misleading on imbalanced data. F1 balances precision and recall; ROC-AUC measures ranking quality.

## Troubleshooting

### Error: `Address already in use` on port 8000

**Cause:** Another process is bound to port 8000.

**Solution:**

```powershell
# Run on a different port
uvicorn app.main:app --port 8001

# Or find and stop the process
netstat -ano | findstr :8000
Stop-Process -Id <PID> -Force
```

### Error: `mlflow.exceptions.RestException`

**Cause:** The MLflow tracking URI is not set or points to an unreachable server.

**Solution:**

```powershell
$env:MLFLOW_TRACKING_URI = "sqlite:///mlruns/mlflow.db"
```

### Error: `422 Unprocessable Entity` on `/predict`

**Cause:** One or more fields are missing, out of range, or have the wrong type.

**Solution:** Check the response body for the specific validation error. Required fields and constraints are defined in `app/schemas.py`.

### Error: `ModuleNotFoundError` when running scripts

**Cause:** The virtual environment is not activated or dependencies are not installed.

**Solution:**

```powershell
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

### Error: PowerShell blocks `Activate.ps1`

**Cause:** The execution policy prevents running activation scripts.

**Solution:**

```powershell
Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned
```

### Error: `mlflow` deprecation warnings during training

**Cause:** The installed MLflow version is newer than the version the code was written against.

**Solution:** The code in `src/train.py` uses the `name=` parameter (MLflow 3.x style). If you are on MLflow 2.x, change it to `artifact_path=`. Alternatively, upgrade MLflow with `pip install --upgrade mlflow`.

### Error: Streamlit shows "Cannot reach the API"

**Cause:** The FastAPI backend is not running or the URL is wrong.

**Solution:** Start the API on the port configured in the Streamlit sidebar (default: 8000). Override with the `API_URL` environment variable if needed.
